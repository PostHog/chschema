# hclexp TLS Support & Container Image â€” Design Spec

**Date:** 2026-05-27
**Status:** Approved, ready for implementation
**Branch:** `worktree-hclexp-tls-and-container` (based on `main`)

## Motivation

PostHog wants to run `hclexp` as an ArgoCD PostSync hook in the
`posthog/charts` repo to dump live ClickHouse schemas after each
deploy. The chart pairs `hclexp` with `amazon/aws-cli` via an
`emptyDir` shared volume to S3-upload the dump; `hclexp` itself stays
filesystem-only and cloud-agnostic.

This requires three things, none of which exist today:

1. TLS support in `hclexp` â€” production ClickHouse is TLS-only on
   port 9440.
2. A minimal, single-purpose container image carrying just `hclexp`.
3. A GitHub Actions workflow that publishes that image to PostHog's
   ECR.

The work splits across two PRs.

## PR 1 â€” TLS / secure-connection support in hclexp

### Surface

| Form              | TLS on                       | Skip cert verify              |
| ----------------- | ---------------------------- | ----------------------------- |
| Flag              | `-secure`                    | `-tls-skip-verify`            |
| Env var           | `CLICKHOUSE_SECURE=true`     | `CLICKHOUSE_TLS_SKIP_VERIFY=true` |
| `clickhouse://` URL query | `?secure=true`        | `?skip-verify=true`           |

- Both flags default `false` â†’ behavior is unchanged for existing
  invocations.
- The URL scheme stays `clickhouse://`. TLS is opted in via query
  param; `?skip-verify=true` without `?secure=true` is an error
  (silent-ignore is a footgun).
- Flag names are stable once chosen â€” the chart in `posthog/charts`
  will reference them.

### Implementation

**`config/clickhouse.go`**
- Extend `ClickHouseConfig` with `Secure bool` and `TLSSkipVerify bool`.
- `GetDefaultConfig()` reads the two new env vars (bool parsing
  consistent with how `CLICKHOUSE_PORT` parses ints today). Empty/unset
  â†’ `false`; `true`/`1`/`yes` â†’ `true`; anything else â†’ `false` plus a
  warn-level log.
- `NewConnection()`: when `cfg.Secure` is true, set
  `options.TLS = &tls.Config{InsecureSkipVerify: cfg.TLSSkipVerify}`
  using `crypto/tls` from stdlib. When false, leave `options.TLS` nil
  â€” preserves current plaintext behavior bit-for-bit.

**`cmd/hclexp/hclexp.go`**
- `runIntrospect` (and any other command that opens a connection via
  flat flags) gets two new flags:
  ```go
  secure := fs.Bool("secure", cfg.Secure, "connect to ClickHouse over TLS")
  skipVerify := fs.Bool("tls-skip-verify", cfg.TLSSkipVerify,
      "skip TLS certificate verification (only valid with -secure)")
  ```
  Defaults come from `cfg` so env vars work without explicit flags.
- After `fs.Parse`, validate: if `*skipVerify && !*secure` â†’ error and
  exit non-zero. Apply to `cfg.Secure` / `cfg.TLSSkipVerify`.

**`loadSide` (diff URL parser)**
- Continue accepting `clickhouse://user:pass@host:port/db1,db2`.
- After URL parse, inspect `u.Query()`:
  - `secure=true` â†’ mark side as secure.
  - `skip-verify=true` â†’ mark side as skip-verify.
  - Same cross-validation: skip-verify requires secure.
- Thread these into the existing `ClickHouseConfig` build path used
  for the live-introspection side.

### Tests

- **Unit (`config/clickhouse_test.go`, new file)**:
  - Each (secure Ă— skip-verify) combo produces the expected `TLS`
    field on the `clickhouse.Options`. Use a constructor that returns
    `*clickhouse.Options` without dialing (`buildOptions(cfg)`), test
    that.
  - Env-var parsing (`true`, `1`, `false`, empty, garbage).
- **Unit (`cmd/hclexp/hclexp_test.go` extension)**:
  - URL â†’ config: plaintext, `?secure=true`, `?secure=true&skip-verify=true`,
    and the error case `?skip-verify=true` without `secure=true`.
- **Live (`docker-compose.yml` + new `test-live-tls` recipe)**:
  - Add a second ClickHouse listener on port 9440 with a self-signed
    cert (generated by an init step or shipped under `test/tls/`).
  - New `justfile` target `test-live-tls` that runs the existing
    introspection test against the secure port with
    `-secure -tls-skip-verify`.
  - Gated behind `ENABLE_CLICKHOUSE` like the existing live tests.
  - **Open question for the user** during impl: do we add this to CI
    in this PR or leave it as a local-only verification? Default
    assumption is **local-only first** to keep CI deterministic;
    promote in a follow-up.

### Docs

- New section "TLS / secure connections" in the README (the one with
  the existing CLI flag docs â€” likely `docs/README.hcl.md`; confirm
  during impl).
- Flag table updated to include `-secure` and `-tls-skip-verify`.
- A short worked example for the URL form.

## PR 2 â€” Container image + publish workflow

### PR 2a â€” Dockerfile + .dockerignore

**`Dockerfile`** (multi-stage, two stages):

```dockerfile
FROM golang:1.26-alpine AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build \
    -trimpath -ldflags="-s -w" \
    -o /out/hclexp ./cmd/hclexp

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /out/hclexp /hclexp
USER nonroot:nonroot
ENTRYPOINT ["/hclexp"]
```

- Builder pinned to `golang:1.26-alpine` (matches CI `go-version`).
- `CGO_ENABLED=0` so the binary works in distroless.
- `-trimpath -ldflags="-s -w"` for reproducibility + smaller image.
- Final stage is `gcr.io/distroless/static-debian12:nonroot` â€” runs
  as UID 65532, no shell, CA roots bundled (public-CA TLS works out
  of the box; `-tls-skip-verify` covers internal CAs).

**`.dockerignore`** â€” exclude everything not needed for the build:

```
.git
.github
.claude
.worktrees
docs
test/snapshots
logs
*.md
*.sql
chschema
hclexp
Dockerfile
.dockerignore
```

Keep Go source, `go.mod`/`go.sum`, `gen/` (frozen protobuf), `config/`,
`internal/`, `cmd/`. Keep `compile-proto.sh` out (legacy).

**Output-to-directory contract**
- Pin the chart's integration shape: `hclexp introspect -out <dir>`
  writes `<dir>/<db>.hcl` per database. Already implemented; add a
  test (`test/introspect_outdir_test.go` or extend existing) that
  asserts this layout against the live cluster so we don't regress it.

**Smoke test**
- A new CI job `docker` that `docker build`s the image and runs
  `docker run --rm <img> -help`, asserting exit 0 and usage output.
- Lives in `.github/workflows/ci.yml`; runs on every push/PR like the
  other jobs.

### PR 2b â€” Publish workflow

**New file `.github/workflows/publish.yml`.**

```yaml
name: Publish
on:
  push:
    branches: [main]
    tags: ['v*']

permissions:
  contents: read
  id-token: write   # required for OIDC

jobs:
  publish:
    runs-on: depot-ubuntu-24.04
    steps:
      - uses: actions/checkout@<pinned-sha>
      - uses: aws-actions/configure-aws-credentials@<pinned-sha>
        with:
          role-to-assume: arn:aws:iam::795637471508:role/<publisher-role-TBD>
          aws-region: us-east-1
      - uses: aws-actions/amazon-ecr-login@<pinned-sha>
      - uses: docker/setup-qemu-action@<pinned-sha>
      - uses: docker/setup-buildx-action@<pinned-sha>
      - id: meta
        uses: docker/metadata-action@<pinned-sha>
        with:
          images: 795637471508.dkr.ecr.us-east-1.amazonaws.com/posthog/chschema
          tags: |
            type=sha,format=short,prefix=sha-
            type=raw,value=latest,enable={{is_default_branch}}
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=semver,pattern={{major}}
      - uses: docker/build-push-action@<pinned-sha>
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

- All actions pinned by SHA, matching the `actions/checkout@93cb...`
  style already used in `ci.yml`.
- `linux/amd64,linux/arm64` because PostHog clusters mix both.
- `id-token: write` permission is required for OIDC role assumption.

**Tags emitted**
- `main` push â†’ `sha-<short>` plus `latest`.
- `vX.Y.Z` tag â†’ `X.Y.Z`, `X.Y`, `X`.

**Open item for the user**
- The publisher role ARN â€” the IAM role this workflow assumes via
  OIDC to push to ECR `795637471508.dkr.ecr.us-east-1.amazonaws.com/posthog/chschema`.
  Will leave a `<publisher-role-TBD>` placeholder and call it out in
  the PR description. Distinct from the consumer-side
  `ch-schema-dump-<env>` IRSA role.

## Cross-Cutting

### Branch / PR strategy
1. PR 1 (TLS) lands first â€” independent, no infra changes.
2. PR 2a (Dockerfile + smoke test) lands second.
3. PR 2b (publish workflow) lands third; once green, the first image
   tag in ECR is comment-backed to `posthog/charts` so the consuming
   chart can wire it up.

Items 2a and 2b may merge as a single PR if review prefers; the
artifact-shape question (image works) is independent of the
ECR-credential question (workflow auth), but they're small enough to
combine.

### Verification before merge
- Each PR needs: `go test ./internal/... ./test/...` green, plus the
  new tests it adds.
- PR 2a additionally: `docker build` + `docker run --rm <img> -help`
  locally and in CI.
- PR 2b: dry-run the workflow on a feature branch first (no `push`
  trigger), then merge.

### Known open items for the user
1. Publisher OIDC role ARN â€” placeholder in workflow.
2. Whether to add the new TLS live-test to CI immediately or keep it
   local-only. Default assumption: local-only.

## Out of scope
- AWS CLI inside the image â€” explicitly not bundled.
- Any S3 / cloud upload logic in `hclexp` â€” that's the uploader
  sidecar's job.
- Replacing the `clickhouse://` scheme name (chart references will
  use the existing scheme + query params).
